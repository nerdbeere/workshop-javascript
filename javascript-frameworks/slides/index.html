Small Overview

Presentation by Béla Varga / @netzzwerg

Frameworks because

  • Abstract away browser differences. (jquery)
  • Add missing javascript core features. (underscore)
  • Toolkit of missing functionality. (sylvester for vector math)
  • App structuring and organizing. (MV*)
  • Rich, Modern UI Widgets.
Modern JavaScript developers realise an MVC framework is essential for managing the complexity of their apps. TodoMVC is a fabulous community contribution that helps developers compare frameworks on the basis of actual project code, not just claims and anecdotes.
Michael Mahemoff

Frameworks selection

  • Many of the are very similar.
  • Classify it into groups.
  • Strong criteria are very important.
  • Don’t be afraid to experiment.

Framework criterias

  • Your project requirements!
  • Framework feature set (Internationalization maybe?).
  • Website with progressive enhancement or big application?
  • Mature and been proved in production?
  • Performance and filesize.

Framework criterias

  • Modular building, modules and testing.
  • Browser support.
  • Plays nicely with others?
  • Inspired or depends on other languages (Rails,CoffeeScript,..)
  • Developer community.

Framework criterias

  • Documentation and books.
  • Licensing and source control.
  • Amount of job offers.
  • Your own subjective coding experience!

Polyfill and Shim

A shim that mimics a future API providing fallback functionality to older browsers.
Paul Irish

Toolkits or Libraries - the light Frameworks

  • Easy to learn.
  • Small file size.
  • Basic and specific functionality.
  • Works with existing architecture.

Frameworks - the heavy ones.

  • For single-page and rich javaScript applications.
  • Complete pre-prepared architecture.
  • Some kind of MV* included.
  • Data binding
  • Rich set of UI Widgets.
  • Memory safety

Vapor.js

  • The World's smallest and fastest JavaScript Library!

jQuery

by John Resig

  • Master of DOM manipulation and animation
  • Doesn't provide an application structure.
  • jQuery-esque method chaining.
  • Very big community.
  • Supported by Microsoft.

jQuery vs. MooTools

Where MooTools aims to make JavaScript more fun, jQuery aims to make the DOM more fun.
Aaron Newton

MooTools

by Valerio Proietti

  • Compact JavaScript framework.
  • Robust, modular and easy to extend.
  • Fully objectoriented.
  • Inheritance with Class constructor.

Underscore

by Jeremy Ashkenas and DocumentCloud

  • Low-level JavaScript utility.
  • Delete your utils file.
  • Uses native functions when available.
  • No extension of built-in JavaScript objects.
  • Some Underscore functionality goes the ECMAScript spec.
  • jQuery and Underscore go hand in hand.

Dojo toolkit

  • One of the first Web applications frameworks.
  • An open collection of tools.
  • UI widgets.
  • Templating.
  • Testing Framework.

Dojo toolkit

  • Own build system.
  • Mature widget infrastructure.
  • AMD compatible.

ExtJS

by Sencha

  • Rich library of cross browser compatible widgets.
  • Component Model with extensions and plugins.
  • Object orienten and full with design patterns.
  • Used in commercial back-office applications.
  • Steeper learning curve.
  • Difficult and restrictive licensing.

Backbone

by Jeremy Ashkenas and DocumentCloud

  • Most well known rock star.
  • Flexible and minimalist solution.
  • MVC (more MVP or MV*).
  • REST-persistable models.
  • Simple routing.

Backbone

  • No automatic data binding
  • Without UI Widgets
  • Templating with Underscore
  • In production for nearly 2 years now

Backbone and Ember

Disclaimer: as software professionals, we must deal with diversity of opinion. Backbone and Ember are results of opinionated and experienced professionals, like you and me. One tool isn't better than the other; they just serve different crowds and, ergo, solve different problems.
Trek

Ember

by Yehuda Katz (formerly of jQuery and Rails)

  • Big framework for rich application building.
  • More magic then backbone.
  • For heavy long time running application.
  • Statemachine-powered hierarchical routing system.
  • Design inspired by Rails and Cocoa.

Ember

  • MVC
  • Live Binding
  • Deferreds
  • Partial View
  • Computed properties. (property that is a result of a function)
  • Handelbars Templating.
  • Still beta.

Meteor

by Meteor development group (foundraised)

  • Radical web development change.
  • Bridges server-side and client-side runtime.
  • Inspired by David Copperfield.

CANjs

by Bitovi

  • The new JavaScriptMVC project.
  • Easy to learn.
  • Small file size.
  • Download builder.
  • Simple routing.
  • Library agnostic

CANjs

  • Basic MVC
  • Live Binding
  • Deferreds
  • Partial View
  • Memory Safety

Knockout

by Knockout team and community

  • Focused on rich UIs
  • Livedata binding
  • Model-View-ViewModel (MVVM) (better than MVC)
  • Library agnostic
  • Partial View

Knockout

  • Observable models with automatic dependency detection.
  • Without routing (use Sammy.js)
  • No own structure for view model.
  • In production for nearly 2 years now

Knockback

by

  • Best of Knockout and Backbone
  • And localization.

Angular

by Google

  • Used internally by Google.
  • Small, weighing in at 60kb
  • Works well with other libraries.
  • Dependency injection.
  • Localization.
  • 2-way data binding.

Angular

  • Fully extensible.
  • Reusable Components.
  • Data binding directly in HTML.
  • Doesn’t need to control whole page.
  • Conceptually a polyfill for missing browser features.

qooxdoo

by 1und1

  • Universal JavaScript framework.
  • Interactive applications (RIAs).
  • Native-like apps for mobile devices.
  • Object-oriented programming model.
  • Theming with CSS3.

qooxdoo

  • HTML 5 Support.
  • Low Level Framework (like jQuery).
  • Mobile App Support.
  • Application-scope environment variables.
  • Built-in unit testing.

Twitter Bootstrap

  • Front-end toolkit for rapidly developing.
  • Collection of CSS and HTML conventions.
  • Simple UI Elements.
  • With 13 custom jQuery plugins.
